                    PROJECT PRESENTATION CONTEXT
================================================================================

Why T2.4?
  Hand-guessed thresholds (160°/70°/85°/140°) work but lack empirical ground.
  Default ROM Threshold pipeline enables data-driven, percentile-based constants.
  These become the starting point for every user's Personal Calibration.

How T2.4?
  Record controlled clips (side + front, multiple quality scenarios).
  Auto-annotate rep boundaries from the angle signal (not manual).
  Compute stats, derive percentiles, validate with replay.

Where T2.4?
  Recording: in-person (user's location)
  Processing: tools/dataset_analysis/ (offline)
  Output: app/lib/core/default_rom_thresholds.dart (shipped)

================================================================================
                            NOTES
================================================================================

1. Start arms fully extended at rest (IDLE state) on EVERY clip.
   This is critical for true rest-angle measurement.

2. Exaggerate "bad form" scenarios intentionally.
   The pipeline uses these for detector validation.

3. Record all clips in one session if possible.
   Consistent lighting/setup reduces per-clip noise.

4. Don't re-record for minor imperfections.
   One weird rep → just label it in quality.
   MediaPipe failure (>10% missing) → re-record.

5. Phase B auto-annotation is the recommended path.
   Manual frame-picking (VLC) is the fallback.

6. Phase E validates thresholds (F1 gate ≥ 0.95).
   If validation fails, investigate which clips are noisy.
   Possible: re-record that one clip and re-run pipeline.
